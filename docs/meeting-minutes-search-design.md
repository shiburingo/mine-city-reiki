# 会議録検索システム設計メモ

## 目的

美祢市Webサイトで公開されている会議録PDFを取り込み、DiscussNetPremiumを手本に、会議録を発言単位で高速検索・閲覧できるようにする。

対象ソース:

- 会議録トップ: `https://www2.city.mine.lg.jp/gyosei/shigikai/11159.html`
- 本会議、常任委員会、特別委員会配下の会議録PDF

## 基本方針

会議録検索は、例規・法令検索の中に「会議録検索システム」タブとして追加する。ただし、PDF解析や会議録特有の判定処理は、例規本文パーサや検索APIへ直書きしない。

同期処理は次の段階に分ける。

1. ソースクロール: 年度、会議種別、会議名、日程、PDF URLを収集する。
2. PDF抽出: PDFからページ、行、座標、文字列、表候補を抽出する。
3. 発言者タグ付け: 発言単位へ分割し、質問者・答弁者・議長・事務局等を判定する。
4. 表整形: PDF中の表を検索可能なテキスト表とWeb表示用テーブルへ変換する。
5. 永続化: 会議、日程、発言、表、資料、抽出ログを保存する。
6. 検索投入: Meilisearchに会議録用インデックスを構築する。

## 発言者タグ付けエンジン

発言者タグ付けは専用プログラムとして分離する。初期実装はルールベースとし、将来は辞書更新、機械学習、LLM補助などに差し替えられるようにする。

### 入力

PDF抽出後の正規化済み行リストを受け取る。

```json
{
  "meeting": {
    "year": 2025,
    "meetingType": "本会議",
    "meetingName": "令和7年第1回定例会",
    "date": "2025-03-13"
  },
  "lines": [
    {
      "page": 1,
      "lineNo": 1,
      "text": "○議長（山田太郎君） ...",
      "x0": 54.1,
      "y0": 120.3
    }
  ],
  "speakerDictionary": {
    "councilMembers": ["..."],
    "executives": ["市長", "副市長", "教育長", "..."],
    "chairAliases": ["議長", "委員長"]
  }
}
```

### 出力

発言単位に分割し、発言者ロールを付ける。

```json
{
  "utterances": [
    {
      "speakerName": "山田太郎",
      "speakerTitle": "議員",
      "speakerRole": "questioner",
      "speakerGroup": "議員",
      "speechType": "question",
      "text": "...",
      "pageStart": 3,
      "pageEnd": 4,
      "confidence": 0.92,
      "reason": "議員名簿一致 + 一般質問見出し配下"
    }
  ],
  "warnings": [
    {
      "code": "unknown_speaker",
      "message": "発言者を確定できません",
      "page": 12,
      "text": "○..."
    }
  ]
}
```

### ロール定義

| role | 意味 |
| --- | --- |
| `questioner` | 質問者。主に議員。一般質問、質疑、討論などを含む。 |
| `answerer` | 答弁者。主に市長、副市長、教育長、部長、課長等の執行部。参考人・証人の発言や、発言冒頭が答弁を示す発言も含む。 |
| `chair` | 議長、委員長など議事進行。 |
| `secretariat` | 事務局、書記など。 |
| `report` | 議事進行から報告・説明を求められた後の報告発言。 |
| `other` | 請願者など、質問・答弁・議事進行・事務局・報告に分類できない発言。 |
| `unknown` | 判定不能。手動確認対象。 |

### 判定ルールの初期案

- `○議長`, `○委員長` は `chair`。
- `○座長` は `chair`。
- `○<氏名>議員`, `○<氏名>君` が議員名簿に一致する場合は `questioner`。
- `○市長`, `○副市長`, `○教育長`, `○総務企画部長`, `○上下水道事業管理者`, `○市立病院事務部事務長` 等は `answerer`。
- `○参考人`, `○証人` は `answerer`。
- 発言本文の冒頭が「お答えします」「質問にお答えします」「説明いたします」等の場合は `answerer`。
- 未分類発言でも、前後に質問者発言があり、本文が質問への説明・回答を示す場合は `answerer`。
- `chair` 発言で報告・説明を求めた直後の説明発言は `report`。
- 一般質問の見出し直下では議員発言を `question`、執行部発言を `answer` とする。
- 質疑、討論、採決などの議事区分は `speechType` で別管理し、検索フィルタに使う。
- 判定に使った辞書・ルールバージョンを保存し、後から再タグ付けできるようにする。

## 表整形エンジン

表整形も専用プログラムとして分離する。目的は「見た目の再現」だけでなく「検索できる表」にすること。

### 入力

PDF抽出段階で得たページ別の文字ボックス、罫線、表候補領域を受け取る。

```json
{
  "page": 18,
  "blocks": [
    {
      "type": "tableCandidate",
      "bbox": [42, 120, 552, 410],
      "cells": [
        { "row": 0, "col": 0, "text": "区分", "bbox": [42, 120, 140, 150] }
      ]
    }
  ]
}
```

### 出力

Web表示用HTML相当の構造と、検索用プレーンテキストを同時に生成する。

```json
{
  "tables": [
    {
      "tableId": "m20250313-p18-t1",
      "caption": "資料要求一覧",
      "page": 18,
      "columns": ["区分", "件名", "担当課"],
      "rows": [
        ["一般質問", "地域交通について", "総合政策課"]
      ],
      "searchText": "資料要求一覧 区分 一般質問 件名 地域交通について 担当課 総合政策課",
      "html": "<table>...</table>",
      "confidence": 0.86
    }
  ],
  "warnings": []
}
```

### 整形方針

- セル結合、改行、縦書き崩れは内部構造で保持し、表示時にHTMLテーブルへ変換する。
- 表の各セルは検索対象に含める。
- 表全体だけでなく、行単位でも検索ヒットできるよう `searchText` を生成する。
- 表抽出に失敗した場合は、崩れた表を捨てず、罫線なしテキストブロックとして保存する。
- エンジンバージョン、抽出信頼度、警告を保存し、後から表だけ再整形できるようにする。

## 推奨モジュール構成

```text
server/
  meeting_minutes/
    crawler.py              # 美祢市会議録ページ巡回
    pdf_extractor.py        # PDF文字・座標・表候補抽出
    speaker_tagger.py       # 質問者/答弁者タグ付け専用
    table_formatter.py      # 表整形専用
    indexer.py              # DB/Meilisearch投入
    models.py               # 入出力データ型
    dictionaries/
      speakers.mine.json    # 議員・執行部・役職辞書
      rules.v1.json         # タグ付けルール
```

## DB設計案

| table | 役割 |
| --- | --- |
| `meeting_sessions` | 会議単位。年度、会議種別、会議名、会期、ソースURL。 |
| `meeting_days` | 日程単位。開催日、PDF URL、PDFハッシュ、抽出状態。 |
| `meeting_speakers` | 発言者辞書。氏名、役職、所属、議員/執行部区分、任期。 |
| `meeting_utterances` | 発言単位。発言者、role、speechType、本文、ページ範囲。 |
| `meeting_tables` | 表単位。HTML、行列JSON、検索用テキスト、ページ、信頼度。 |
| `meeting_extract_runs` | 抽出実行履歴。エンジンバージョン、件数、警告、エラー。 |

## Meilisearch設計案

会議録は例規・法令とは別インデックスにする。

- `mine_city_meeting_minutes`
- 文書種別: `session`, `day`, `utterance`, `table`
- 主検索対象: 発言本文、表検索テキスト、発言者名、会議名、日程
- フィルタ: 年度、開催日、会議種別、発言者、speakerRole、speechType
- ソート: 開催日、ヒット順

## 初期実装の優先順位

1. 会議録PDFリンクのクロールとDB保存。
2. PDFテキスト抽出と発言単位分割。
3. 発言者タグ付けエンジンv1。
4. 会議録検索UIタブ追加。
5. 表整形エンジンv1。
6. DB検索による初期検索API。
7. 会議録専用Meilisearch投入、詳細検索、検索履歴、発言集作成。

表は会議録検索の品質に影響するため、初期リリースで最低限「検索可能なテキスト表」として保存し、HTML表の高精度再現は段階的に改善する。

2026-06時点の初期版では、直近1年分のPDFを対象に、DB検索でキーワード・発言者・質問者/答弁者・会議種別フィルターを提供する。検索結果は該当発言に加えて前後発言を返し、質問と答弁を同じ画面で確認できる。
